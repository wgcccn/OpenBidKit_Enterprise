# 目录生成修复待办

## 背景

Step03 目录生成目前仍存在一类固定流水线问题：模型在首次审核阶段已经给出具体、局部、可执行的修改建议，但程序没有基于当前目录做定向修复，而是走“重新提取评分大类 + 全量重新生成目录”的路径。

该问题会导致：

- 首次审核建议没有被当成 patch 指令使用。
- 已经较好的目录结构被整体重写，增加不确定性。
- AI token、耗时和失败概率上升。
- 修复目标不够聚焦，容易把局部问题扩大成全局重生成。

## 已观察到的日志案例

日志时间：2026-06-18 10:09 至 10:10。

相关日志文件位于本机开发者 AI 日志目录：

- `2026-06-18T10-09-42-320Z-首次审核-*.json`
- `2026-06-18T10-09-59-172Z-技术评分大类-*.json`
- `2026-06-18T10-10-23-070Z-最终审核-*.json`

首次审核返回的问题是局部问题：

```text
R2（项目投入人员团队）下的 2.3（人员稳定性保障措施）内容与 R6（人员稳定性保障措施）重复，且 R2 细项中不包含人员稳定性，建议删除 2.3 或将相关内容整合至 R6 中。
```

理想处理方式：

- 识别这是局部重复/归属错误。
- 删除或迁移 `2.3`。
- 如 R6 覆盖不足，仅在 R6 下补充必要目录。
- 重新编号。
- 再次复核。

当前实际处理方式：

- `alignedWorkflow()` 收到 `passed=false`。
- 将 `suggestions` 传给 `extractRequirementGroups()`。
- 重新提取技术评分大类。
- 调用 `buildAligned()` 全量重新生成目录。

对应代码位置：

- `client/electron/services/outlineGenerationTask.cjs`
- `alignedWorkflow()`
- 当前二次处理逻辑为“审核失败 -> 重新提取评分大类 -> 全量重新生成”。

## 问题本质

当前目录生成流程是固定流水线，缺少“审查 -> 反思 -> 定向修复 -> 复核”的 agentic 修复能力。

模型已经提供了局部诊断，但程序没有可复用的 patch 工具和 agent loop 来执行局部修复。

## 已完成的相关修复

以下问题已经完成代码修复，本待办不再重复处理：

- 下级目录补充修复 Prompt 已增加返回格式示例，避免弱模型把可恢复内容修成空 `additions`。
- 扩写目录流程已调整为评审前不因目录未完整提前中断，最终评审修复后再硬校验。
- `validateCompleteOutline()` 已改为“完整目录整体至少包含三级结构”，不再要求每个一级目录都必须有三级。
- 扩写流程中一级补充计划失败、单个章节下级补充失败时，会保留当前目录继续进入最终评审修复。

## 待办目标

本问题暂存，后续集中在 AI Agent Runtime 中解决。

目标不是再写一个特殊分支，而是引入可复用的目录修复 agent：

- AI 先分析审核建议属于哪类问题。
- AI 决定是否局部修复、重新生成单个章节、重新提取评分大类或结束。
- 程序只允许 AI 通过受控工具修改目录。
- 最终结果必须通过程序硬校验后才能保存。

## 推荐后续实现方向

第一阶段优先改造普通技术方案 `alignedWorkflow()`：

1. 首次审核失败后不默认全量重生。
2. 调用目录修复 Agent。
3. Agent 可使用工具：读取当前目录、生成 patch、应用 patch、复核目录、生成单个一级目录下级目录。
4. 只有当 Agent 判断“评分大类提取错误”时，才允许重新提取评分大类。
5. 最多执行 2-3 轮定向修复。
6. 最终仍失败时，保留最后一版目录和审核问题，避免无目的全量重写。

## 验收标准

- 对“某个二级目录放错评分大类”这类问题，只做局部删除/迁移/补充，不触发全量重生。
- 对“某个评分大类缺少关键细项”这类问题，只重生成或补充对应一级目录下的子目录。
- 对“一级目录数量或标题与评分大类不匹配”这类问题，才允许重新提取评分大类或重建一级目录。
- 每一轮修复都有日志记录：问题、动作、影响节点、复核结果。
- 最终保存前仍执行程序硬校验。
