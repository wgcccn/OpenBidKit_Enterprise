# 提示词梳理

本文只基于当前已阅读代码整理，不做效果评价和优化建议。凡是固定提示词，按代码原文写出；凡是运行时插入内容，用中文变量占位。

## 范围

- Main 侧真实 AI 请求：`electron/services/aiService.cjs`、`bidAnalysisTask.cjs`、`outlineGenerationTask.cjs`、`globalFactsTask.cjs`、`contentGenerationTask.cjs`、`knowledgeBaseService.cjs`、`rejectionCheckTask.cjs`、`agentService.cjs`、`electron/utils/segmentedAiResultMerger.cjs`。
- Renderer 侧仍存在的测试/旧链路：`src/features/settings/pages/SettingsPage.tsx`、`src/features/developer/pages/DeveloperTestPage.tsx`、`src/features/developer/pages/OpenCodeAgentTestPage.tsx`、`src/features/technical-plan/services/outlineWorkflow.ts`、`src/shared/prompts/*`。
- 普通 HTTP 请求，例如更新检查、GitHub Release 请求，不属于 AI 请求，不列入。

## 公共变量说明

- 【完整招标文件 Markdown】：从工作区读取的招标文件 Markdown；如果招标文件被切分，则这里是当前分段 Markdown。
- 【标段上下文提示】：用户选择标段后由 `buildBidSectionContextHint()` 基于 AI 标段识别结果生成的 system 消息；找不到具体标段对象时只注入多标段已按用户选择范围处理的标志位提示。
- 【项目概述】：Step02 `projectOverview` 成功结果或目录数据中的项目概述。
- 【技术评分要求】：Step02 `techRequirements` 成功结果。
- 【Step02 关键解析结果】：项目信息、甲方信息、交货和服务要求三项成功结果拼接。
- 【招标文件关键信息】：项目概述 + Step02 关键解析结果。
- 【目录树】：当前技术方案目录 JSON 或格式化目录文本。
- 【全局事实变量】：Step04 保存的 `globalFacts`。
- 【参考知识库条目】：知识库读取出的标题、摘要或正文，取决于调用点。
- 【当前小节】：当前叶子目录节点的编号、标题、描述、上级章节和同级章节。
- 【当前小节正文】：当前小节已生成或待审计/待修复的正文。
- 【模型返回的无效 JSON】：AI 主请求返回但解析或校验失败的内容，最多截取 60000 字符。

```text
你是一个严格的 JSON 修复助手。请根据给出的原始内容和校验问题，修复现有结果。

要求：
1. 优先在原结果基础上做最小必要修改，不要整体重写
```

```text
程序最终只读取 original-outline.json 文件内容，请确保该文件是可被 JSON.parse 直接解析的纯 JSON。

      "description": "目录说明",
      "children": [
        {
          "id": "1.1",
          "title": "二级目录标题",
          "description": "目录说明",

              "id": "1.1.1",

7. 任务结束时，original-outline.json 就是可供程序使用的旧目录 JSON。
```

输入文件顺序：

1. `original-plan.md`：【原方案 Markdown】

```text
    {
```

### 技术方案目录修复 Agent

调用点：`electron/services/outlineGenerationTask.cjs` 的 `buildOutlineAgentRecoveryPrompt()`。

触发条件：评分大类提取、评分项对齐目录生成、最终目录审核或程序校验失败，并且错误属于可恢复类型。

Prompt：

```text
你是易标投标工具箱中的目录生成 Agent。请在当前工作目录中读取输入文件，自主修复技术标目录，并把最终结果写入 【输出文件名】。

当前目录生成模式：【普通技术方案目录生成 / 已有方案扩写-仅使用原方案目录 / 已有方案扩写-AI补充目录】

【如果有恢复触发原因：本次恢复触发原因：【原因】】

最终目标：
生成一份可以直接保存为技术方案目录的 JSON，目录必须覆盖技术评分要求，并解决 final-review.json 中指出的问题；如果 current-outline.json 为空或不完整，需要直接生成完整目录。

工作方式：
1. 先阅读 project-overview.md、technical-requirements.md、current-outline.json、final-review.json 和 workflow.json。
2. 如果存在 requirement-groups.json，必须同时阅读并保持一级目录与 groups 对齐；如果你判断 groups 本身提取错误，可以自行修正 groups 并同步修正一级目录。
3. 如果不存在 requirement-groups.json，普通技术方案目录生成模式下需要先从 technical-requirements.md 提取 groups，并同步生成与 groups 对齐的一级目录。
4. 如果存在 original-outline.json，请优先在原目录基础上做必要修复，不要无目的全量重写。
5. 修复应以定向处理为主，包括删除重复项、迁移错位目录、补充缺失目录、合并明显重复目录和重新编号。
6. 最终自行复核：目录必须满足 workflow.json 中的 hard_constraints。

输出要求：
1. 必须把最终结果写入 【输出文件名】。
2. 【输出文件名】 必须是纯 JSON，不要包含 Markdown 代码块或解释文字。
3. JSON 顶层格式必须为：
【普通技术方案：包含 groups 和 outline；已有方案扩写：只包含 outline】
4. 不要输出正文 content、图片、表格、Mermaid、审查说明或额外字段。
5. 编号可以自行整理，程序会再次统一编号；但层级关系必须正确。
```

输入文件顺序：

1. `project-overview.md`：【项目概述】
2. `technical-requirements.md`：【技术评分要求】
3. `workflow.json`：【目录生成模式、恢复类型、硬约束】
4. `current-outline.json`：【当前目录 JSON】
5. `final-review.json`：【最终审核结果或合成审核结果】
6. 可选 `requirement-groups.json`：【技术评分大类 JSON】
7. 可选 `original-outline.json`：【原方案目录 JSON】
