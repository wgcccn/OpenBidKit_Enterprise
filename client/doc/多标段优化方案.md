# 多标段优化方案

## 背景

当前技术方案流程在 Step01 上传招标文件后，通过正则规则识别多标段。如果识别到多个标段，会让用户选择本次投标范围。用户选择后，系统仍保存完整招标文件 Markdown，后续 Step02、Step04 通过额外注入“只关注所选标段”的 system prompt 来约束 AI。

这种方式实现简单，但存在几个问题：

1. 正则只能覆盖标准标题格式，无法稳定识别表格、清单、采购范围说明里的标段信息。
2. 标段选择发生在 Step01 上传阶段，会打断“上传文件”这个单一职责。
3. 后续 AI 仍读取完整招标文件，其他标段的采购清单、评分项、投标保证金等内容仍可能污染解析结果。
4. 单纯依赖 prompt 约束无法保证模型完全忽略其他标段内容。

## 目标

把多标段处理从“上传后正则识别 + system prompt 约束”升级为“Step02 用户选择模式 + AI 提取标段范围 + 生成投标范围工作副本”。

升级后应满足：

1. Step01 只负责上传和解析 Markdown，不再处理标段选择。
2. Step02 开始解析前，由用户主动选择单标段或多标段模式，默认单标段。
3. 正则识别保留，但只用于校验和提示，不作为最终标段列表来源。
4. 多标段模式下，由 AI 提取标段列表和每个标段的原文范围。
5. 用户选择本次投标标段后，Main 侧生成投标范围工作副本。
6. 后续 Step02、Step03、Step04、Step05 都基于工作副本执行。
7. 原始招标文件 Markdown 完整保留，便于重新选择标段、恢复、排查和重新解析。

## 总体流程

1. Step01 上传招标文件，只解析 Markdown，不弹出标段选择。
2. Step02 解析设置页增加“投标范围模式”：单标段、多标段。
3. 用户点击“开始解析”后，先运行现有正则规则做快速校验。
4. 用户选择单标段但正则发现疑似多标段时，提示用户“切换多标段解析”或“继续单标段解析”。
5. 用户选择多标段但正则未发现多标段时，提示用户“继续 AI 识别”或“改为单标段解析”。
6. 多标段模式下，先启动 AI 标段提取任务。
7. AI 返回标段列表、每个标段的行号范围和证据。
8. 用户选择本次要投的标段。
9. Main 侧基于原始 Markdown 生成投标范围工作副本。
10. Step02 后续解析项基于工作副本执行，同时继续注入所选标段提示。
11. Step03、Step04、Step05 继续读取 `readTenderMarkdown()`，实际读取到的是工作副本。

## 数据设计

### 文件

保留两份招标文件 Markdown：

1. `technical-plan/tender-original.md`

   原始招标文件 Markdown，完整保留。

2. `technical-plan/tender.md`

   当前解析使用的投标范围工作副本。

单标段模式下，`tender-original.md` 和 `tender.md` 内容一致。

多标段模式下，`tender-original.md` 是完整原文，`tender.md` 是剔除其他明确标段内容后的工作副本。

### SQLite 字段

在 `technical_plan_meta` 中建议新增字段：

| 字段 | 说明 |
| --- | --- |
| `bid_section_mode` | 投标范围模式，取值 `single` 或 `multiple` |
| `bid_sections_json` | AI 提取出的标段列表和行号范围 |
| `bid_section_extraction_status` | 标段提取状态，例如 `idle`、`running`、`success`、`error` |
| `bid_section_extraction_error` | 标段提取失败原因 |
| `tender_original_markdown_path` | 原始招标文件 Markdown 相对路径 |
| `tender_original_markdown_hash` | 原始招标文件内容 hash |
| `tender_original_markdown_chars` | 原始招标文件字符数 |

继续复用现有字段：

| 字段 | 说明 |
| --- | --- |
| `tender_markdown_path` | 后续流程权威输入，指向工作副本 |
| `tender_markdown_hash` | 工作副本内容 hash |
| `tender_markdown_chars` | 工作副本字符数 |
| `selected_section_id` | 用户选择的标段 ID |
| `selected_section_title` | 用户选择的标段标题 |
| `selected_section_head_line` | 用户选择的标段原文标题行 |

`tender_markdown_path` 继续作为后续流程权威输入，降低 Step02 后续任务、废标项导入、导出等调用方改动范围。

## 类型设计

建议扩展技术方案类型：

```ts
export type BidSectionMode = 'single' | 'multiple';

export type BidSectionExtractionStatus = 'idle' | 'running' | 'success' | 'error';

export interface BidSectionLineRange {
  startLine: number;
  endLine: number;
  reason?: string;
}

export interface AiDetectedBidSection {
  id: string;
  index: number;
  unit: string;
  title: string;
  headLine: string;
  description: string;
  includeRanges: BidSectionLineRange[];
  evidence: string[];
}
```

`TechnicalPlanState` 建议增加：

```ts
bidSectionMode: BidSectionMode;
bidSections: AiDetectedBidSection[];
bidSectionExtractionStatus: BidSectionExtractionStatus;
bidSectionExtractionError?: string;
```

## AI 标段提取

### 输入策略

1. Main 侧读取原始 Markdown。
2. 给 Markdown 加全局行号，例如 `L000123 | 原文内容`。
3. 长文档按上下文限制分段，但必须保留全局行号。
4. 每段分别提取标段候选。
5. 合并所有分段候选，形成最终标段列表。

### 输出格式

AI 只返回结构化 JSON，不直接输出裁剪后的 Markdown。

建议输出：

```json
{
  "totalDeclared": 2,
  "sections": [
    {
      "id": "section-1",
      "index": 1,
      "unit": "标段",
      "title": "一标段",
      "headLine": "一标段：设备采购及安装",
      "description": "设备采购、安装、调试及售后服务。",
      "includeRanges": [
        { "startLine": 120, "endLine": 180, "reason": "一标段采购清单" }
      ],
      "evidence": ["一标段：设备采购及安装"]
    }
  ]
}
```

### Prompt 约束

Prompt 必须明确：

1. 只识别招标文件中明确属于某个标段、标包、分包、采购包、包件、标的的内容。
2. 通用条款不要归入某个标段。
3. 只有明确属于某个标段的连续内容才输出 `includeRanges`。
4. 不确定归属的内容不要输出范围，默认作为通用内容保留。
5. 不要编造标段，不要补写原文没有的范围。
6. 行号必须来自输入文本，不允许超出输入行号范围。
7. 返回 JSON，不要输出 Markdown、代码块或解释。

### Main 侧校验

Main 侧需要对 AI 结果做基础校验：

1. `sections` 必须是数组。
2. `includeRanges` 必须在原始 Markdown 行数范围内。
3. `startLine <= endLine`。
4. 标段标题为空的候选丢弃。
5. 没有任何有效标段时，标段提取任务失败并提示用户。

## 工作副本生成

用户选择标段后，Main 侧基于原始 Markdown 生成 `technical-plan/tender.md`。

生成规则：

1. 读取原始 Markdown 的所有行。
2. 收集选中标段的 `includeRanges`。
3. 收集其他标段的 `includeRanges`。
4. 删除“属于其他标段且不属于选中标段”的行。
5. 保留所有未被 AI 明确标记为其他标段的内容。
6. 保留所有通用内容，例如投标人资格、投标文件格式、评标办法通用规则、合同条款、开标流程。
7. 如果某段同时被标记为选中标段和其他标段，保留该段，避免误删。
8. 写入 `technical-plan/tender.md`。
9. 保存 `selected_section_*` 字段。
10. 清空 Step02 已解析结果、目录、全局事实、正文缓存。

不要让 AI 直接输出裁剪后的 Markdown。AI 只负责输出结构化范围，实际删除由程序按行号执行。

## Step02 UI 改造

### 设置页

在 `BidAnalysisPage` 设置弹窗顶部增加“投标范围”配置区。

配置项：

1. 单标段

   默认选项。按完整招标文件解析。

2. 多标段

   先识别标段并选择本次投标范围，再解析招标文件。

### 状态展示

配置 chip 建议显示：

1. 单标段：`单标段 · 5 项`
2. 多标段未选择：`多标段 · 待选择 · 5 项`
3. 多标段已选择：`多标段 · 一标段 · 5 项`

### 启动解析

用户点击“开始解析”时：

1. 保存解析项配置和投标范围模式。
2. 运行正则校验。
3. 根据模式和校验结果决定是否弹提示。
4. 单标段模式直接启动原 Step02 解析任务。
5. 多标段模式且未提取标段时，先启动 AI 标段提取任务。
6. 标段提取完成后弹出标段选择弹窗。
7. 用户选择标段后生成工作副本。
8. 自动继续启动原 Step02 解析任务。

### 恢复

如果应用关闭后恢复到多标段待选择状态，再次进入 Step02 时应继续显示标段选择弹窗。

如果标段提取任务失败，设置页显示失败原因，并允许用户重新提取或切回单标段。

## Main 侧任务设计

### 新增任务类型

新增 `bid-section-extraction` 后台任务，归属 `technical-plan` 任务组。

职责：

1. 读取原始招标文件 Markdown。
2. 生成带行号文本。
3. 调用 AI 提取标段候选。
4. 合并分段结果。
5. 写入 `bid_sections_json` 和 `bid_section_extraction_status`。
6. 通过任务事件推送进度。

### 选择标段 IPC

用户选择标段不需要长任务，只需要一次 IPC：

1. 读取 `bid_sections_json`。
2. 根据选中标段生成工作副本。
3. 写入 `tender.md`。
4. 保存 `selected_section_*`。
5. 清空 Step02 和后续结果。
6. 返回最新 `TechnicalPlanState`。

### 原 Step02 解析任务

原有 `bid-analysis` 任务继续负责项目概况、技术要求、评分项等解析。

多标段模式下，启动 `bid-analysis` 前必须满足：

1. 已完成标段提取。
2. 已选择标段。
3. 已生成工作副本。

单标段模式下，`bid-analysis` 直接使用完整工作副本。

## 正则校验逻辑

现有 `detectBidSections()` 保留，但定位调整为“校验提示”。

规则：

1. 用户选单标段，正则识别到多个标段时，提示用户可能是多标段。
2. 用户选多标段，正则未识别到多个标段时，提示用户可能不是多标段。
3. 用户可以选择切换模式，也可以继续当前选择。
4. 正则结果不写入最终标段列表。
5. 最终标段列表以 AI 提取结果为准。

## 后续流程影响

### Step02 招标文件解析

Step02 继续调用 `workspaceStore.readTenderMarkdown()`，读取工作副本。

多标段模式下，继续保留 `sectionHint` 注入，作为额外约束，防止工作副本仍残留跨标段描述。

### Step03 目录生成

不需要直接读取标段状态。

目录生成基于 Step02 解析结果和工作副本链路自然收敛到所选标段。

### Step04 全局事实

继续读取工作副本，并继续注入 `sectionHint`。

全局事实应只围绕所选标段和通用条款生成。

### Step05 正文生成

不需要直接读取标段状态。

正文生成依赖 Step02 解析结果、Step03 目录、Step04 全局事实。

### 废标项检查导入

从技术方案导入招标文件时，会导入工作副本，行为与当前投标范围一致。

## 状态清空规则

1. 替换招标文件时，清空标段提取结果、选中标段、工作副本、Step02 结果和所有后续结果。
2. 从单标段切换到多标段时，清空 Step02 结果和后续结果。
3. 从多标段切换到单标段时，清空选中标段、标段提取结果、Step02 结果和后续结果，并把工作副本重置为原始 Markdown。
4. 多标段下重新选择标段时，重新生成工作副本，并清空 Step02 结果和后续结果。
5. 只调整 Step02 解析项，不改变标段模式和选中标段时，沿用现有清空策略。

## 兼容迁移

1. 旧版本只有 `tender.md`，没有 `tender-original.md`。
2. 升级后如果缺少原始文件路径，则把现有 `tender.md` 视为原始文件和工作副本。
3. 旧版本已有 `selected_section_*` 时保留显示，但不会自动生成过滤副本。
4. 用户重新选择多标段解析后，才通过 AI 生成新的工作副本。
5. 旧的 Step01 pending 标段选择状态在升级后清理，不再恢复 Step01 弹窗。

## 实施步骤

1. 扩展 `TechnicalPlanState`、IPC 类型和相关前端类型。
2. 扩展 SQLite schema 和 runtime migration。
3. 改造 `technicalPlanStore.importTenderDocument()`，取消上传阶段正则标段选择。
4. 增加原始 Markdown 和工作副本读写能力。
5. 新增 AI 标段提取任务和 Prompt。
6. 新增工作副本生成工具，按行号范围删除其他标段内容。
7. 改造 Step02 设置页，增加单标段、多标段配置和正则校验提示。
8. 增加标段选择弹窗在 Step02 的恢复和确认逻辑。
9. 改造 `startBidAnalysis` 前置流程，多标段先提取和选择，再进入原解析任务。
10. 保留并复用 `sectionHint` 注入逻辑。
11. 清理 Step01 原有 pending 标段选择 UI 和相关临时文件逻辑。
12. 补充构建验证和手动流程验证。

## 验证清单

1. 单标段文件上传后，Step02 默认直接解析。
2. 单标段模式但正则发现多标段时，会提示用户切换或继续。
3. 多标段模式下，AI 能提取标段列表并弹出选择。
4. 选择标段后，`tender-original.md` 完整保留。
5. 选择标段后，`tender.md` 只删除其他明确标段内容，通用内容保留。
6. 重新选择标段会清空旧 Step02 和后续结果。
7. Step02 解析结果不再混入其他标段采购清单、评分项或投标保证金。
8. Step03、Step04、Step05 能沿用工作副本正常生成。
9. 应用关闭后，多标段识别结果和已选投标范围可从工作区状态恢复。
10. Windows 中文路径和中文文件名正常。
11. 执行 `cd client; npm run build` 通过。

## 风险与取舍

1. AI 标段范围识别可能误判，因此只删除明确归属于其他标段的范围，不确定内容全部保留。
2. 工作副本可能仍残留少量其他标段信息，因此继续保留基于 AI 标段结果生成的标段上下文提示作为二次约束。
3. 多标段提取会增加一次 AI 调用成本，但只在用户明确选择多标段时触发。
4. 保留原始 Markdown 会增加少量本地存储占用，但能显著提升可恢复性和可排查性。

## 结论

该方案相比当前实现更可靠。它把标段识别放到更合适的 Step02，允许用户主动选择投标范围模式，并通过 AI 提取标段范围生成工作副本，从源头减少其他标段内容对后续解析和生成的污染。

实施时应坚持两个原则：

1. 原始招标文件完整保留。
2. AI 只输出结构化行号范围，实际裁剪由程序执行。
