## step 05 正文

正文生成需要在“已有方案扩写”模式下改造。核心目标是：原方案是用户已经写好的技术方案正文，扩写后必须保留原方案中的实质内容，并在此基础上优化和扩充字数。

### 总体原则

- “生成技术方案”保持现有正文生成逻辑不变。
- “已有方案扩写”复用现有 Step05 页面、任务、正文编排、最低字数扩充、一致性审计和配图流程。
- 扩写模式新增“原方案还原”阶段，位置在现有“正文编排”之后、“首轮正文生成”之前。
- 原方案还原阶段不让 AI 改写原文，只让 AI 判断“原方案段落应该归属到哪个当前目录叶子节点”。
- 程序负责按照 AI 返回的段编号拼接原方案原文，并写入对应叶子节点正文。
- 首轮正文生成时，如果叶子节点已经还原了原方案正文，则不再从零编写，而是基于已还原正文做优化和扩写。
- 最低字数扩充阶段保持现有逻辑，不额外提交原方案，因为首轮正文已经基于原文处理过。
- 原方案覆盖审计做成扩写模式专属设置项，默认关闭，用户开启后才执行。

### 正文生成主流程

扩写模式下 Step05 流程为：

1. 启动正文生成任务。
2. 校验已完成目录生成和全文事实设定。
3. 读取原方案 Markdown。
4. 执行现有正文编排阶段。
5. 执行新增“原方案还原”阶段。
6. 执行首轮正文生成。
7. 执行现有最低字数检查、补目录和正文扩写。
8. 如果用户开启“原方案覆盖审计”，执行覆盖审计和修复。
9. 执行现有全文一致性审计。
10. 执行现有配图阶段。
11. 保存最终正文，仍以 `outlineData.outline[*].content` 为权威来源。

### 原方案拆分

程序先把原方案 Markdown 拆成可引用的原文段。

建议优先按 Markdown 标题结构拆分：

```json
{
  "id": "P001",
  "title_path": ["三、项目实施方案", "3.1 实施组织"],
  "content": "原方案中的完整段落或章节内容",
  "hash": "sha256...",
  "chars": 520
}
```

如果原方案标题结构不规范，则按连续段落拆分。

每个原文段必须有稳定编号，例如 `P001`、`P002`、`P003`。

### 原方案还原阶段

还原阶段只做“原文段落归属判断”。

AI 输入：

- 当前完整目录叶子节点清单。
- 每个叶子节点的 ID、标题、描述、上级章节、同级章节。
- 原方案拆分后的段编号、标题路径、正文摘要或正文片段。
- Step02 关键解析结果。
- Step04 全文事实变量标题。

AI 输出只能是段编号映射，不允许输出新正文。

返回格式示例：

```json
{
  "assignments": [
    {
      "node_id": "3.2.1",
      "source_ids": ["P014", "P015"]
    },
    {
      "node_id": "3.2.2",
      "source_ids": ["P016"]
    }
  ]
}
```

程序校验：

- `node_id` 必须是当前目录叶子节点。
- `source_ids` 必须来自原方案段编号。
- 原方案段默认只分配给一个主节点。
- 如果 AI 返回不存在的节点或段编号，走 JSON 修复或直接丢弃非法项。
- 未分配的原方案段需要记录到任务日志，供覆盖审计或人工排查。

程序处理：

- 按 `source_ids` 顺序拼接原方案原文。
- 直接写入对应叶子节点正文。
- 同步更新 `contentGenerationSections[nodeId].content`。
- 同步更新 `outlineData.outline[*].content`。
- 不让 AI 生成所谓“还原正文”，程序只拼接真实原文。

### 编排结果扩展

在每个叶子节点的 `contentGenerationPlans.plan` 中增加原方案还原状态。

字段建议：

```ts
original_material: {
  restored: boolean;
  optimized: boolean;
  source_ids: string[];
  source_titles: string[];
  source_hashes: string[];
  restored_chars: number;
  restored_at?: string;
  optimized_at?: string;
}
```

含义：

- `restored`：当前叶子节点是否已经还原过原方案正文。
- `optimized`：当前叶子节点是否已经基于还原正文完成首轮优化扩写。
- `source_ids`：还原到当前节点的原方案段编号。
- `source_titles`：原方案段标题路径，用于日志和审计。
- `source_hashes`：原方案段内容 hash，用于覆盖审计和变更检测。
- `restored_chars`：程序拼接进来的原文长度。
- `restored_at`：还原时间。
- `optimized_at`：首轮优化扩写完成时间。

普通“生成技术方案”不写入该字段，或写入默认空状态。

### 首轮正文生成分支

首轮正文生成时，按 `original_material.restored` 判断分支。

未还原节点：

- 走现有普通正文生成逻辑。
- 使用现有 `buildChapterContentMessages()`。
- 行为不变。

已还原节点：

- 使用新的扩写分支提示词。
- 输入当前节点已还原的原方案正文。
- 不再从零编写。
- 要求模型在原文基础上优化和扩写。

扩写分支提示词核心要求：

- 已还原正文是用户原方案中的真实内容，是本章节必须保留的基础。
- 必须保留原文中的实质信息、技术路线、服务承诺、设备参数、人员安排、周期、验收、售后、实施方法。
- 可以调整语序、合并重复表达、提升专业性、补充细节、增加过渡和说明。
- 不得删除原文已有关键事实。
- 不要提到“原方案”“历史文档”“用户原文”。
- 输出当前章节完整正文，不输出标题。
- 严禁输出 Mermaid、图片 Markdown、伪目录标题。

还原分支生成完成后：

- 用模型返回的新正文替换当前还原正文。
- 设置 `original_material.optimized = true`。
- 更新 `optimized_at`。
- 保存到 `contentGenerationSections` 和 `outlineData.outline[*].content`。

### 任务续跑规则

现有逻辑会跳过已有正文的小节。扩写模式需要额外判断。

如果节点满足下面条件之一，需要进入首轮生成：

```text
regenerate
section.status === 'error'
content 为空
original_material.restored === true && original_material.optimized !== true
```

这样可以避免任务在“原方案还原完成，但优化扩写未完成”时中断后，继续任务却误认为正文已经完成。

### 补目录限制

最低字数不足时，现有逻辑会自动补目录。扩写模式下必须限制补目录，避免破坏已还原正文节点。

规则：

- 如果某叶子节点 `original_material.restored === true`，该节点禁止新增子节点。
- 补目录上下文里把该节点标记为 `locked-restored`。
- AI 只能选择 `add:*` 节点作为 `parent_id`。
- `locked-restored` 节点不能作为补目录父节点。

目录上下文示例：

```text
- 3.2 | L2 | locked-restored | 运维服务方案
- 3.3 | L2 | add:L3 | 应急保障措施
```

补目录提示词增加规则：

- `locked-restored` 节点已经承载用户原方案正文，严禁新增子节点。
- 不允许把已还原正文节点拆成下级目录。
- 只能围绕未还原、可扩展的节点补充目录。

这样避免出现“还原正文 -> 补目录拆节点 -> 重新还原 -> 再补目录”的循环。

### 字数扩充阶段

字数扩充阶段保持现有逻辑。

原因：

- 首轮正文生成已经基于还原原文完成优化和扩写。
- 后续最低字数扩充只是在当前正文基础上继续扩充。
- 不需要额外提交原方案，否则会增加成本并引入重复内容。

### 原方案覆盖审计

新增设置项：

```ts
enableOriginalPlanCoverageAudit: boolean
```

默认值：

```ts
false
```

显示规则：

- 只在 `workflowKind === 'existing-plan-expansion'` 时显示。
- 普通“生成技术方案”不显示。
- 普通模式不传该配置，也不执行覆盖审计。

UI 文案建议：

```text
原方案覆盖审计
检查原方案中的核心内容是否已经保留到生成正文中。默认关闭，开启后会增加一次审计和修复请求。
```

执行位置：

- 首轮正文生成完成。
- 最低字数扩充完成。
- 原方案覆盖审计执行。
- 然后进入现有全文一致性审计。

覆盖审计输入：

- 原方案段编号和原文。
- 段编号与叶子节点的映射关系。
- 当前叶子节点最终正文。

覆盖审计输出：

```json
{
  "items": [
    {
      "source_id": "P014",
      "node_id": "3.2.1",
      "status": "covered",
      "missing_points": [],
      "repair_suggestion": ""
    },
    {
      "source_id": "P015",
      "node_id": "3.2.1",
      "status": "partial",
      "missing_points": ["缺少原文中的7x24小时响应承诺"],
      "repair_suggestion": "补充7x24小时响应承诺"
    }
  ]
}
```

`status` 取值：

```text
covered
partial
missing
conflict
```

处理规则：

- `covered` 不处理。
- `partial` 和 `missing` 生成局部补丁。
- `conflict` 写入日志，优先不自动覆盖，交给现有一致性审计或人工确认。
- 补丁使用现有 `insert` / `replace` 机制。
- 修复后再进入全文一致性审计，避免补回原方案内容后产生前后冲突。
